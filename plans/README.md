## Recommended execution order

1. **001** and **002** first — both are Performance findings with no dependency on anything else, and the highest leverage (HIGH / MEDIUM-HIGH).
2. **003** and **004** next — independent Accessibility fixes, can run in either order or in parallel with each other.
3. **005** and **006** — small Cohesion cleanups, independent of everything above; safe to batch together since both are single-rule, low-risk edits.
4. **007** and **008** last — additive Missed opportunities, purely new motion, no risk of conflicting with the corrective fixes above.

## Dependencies between plans

- **001 and 007 touch the same component** (`Header.tsx`'s mobile nav / hamburger button) but different, non-overlapping parts: 001 wraps the `<nav>` children in a new `.nav-inner` div and changes `.site-header nav`/`.site-header nav.open` CSS; 007 only changes the `<button className="menu-toggle">` line and `.menu-toggle`/`.menu-toggle span` CSS. They can be executed in either order or by different agents in parallel — if executed by the same agent, do 001 first since it's higher severity, then 007.
- **004 and 008 are loosely related**: 004 defines a reduced-motion opacity-fade override for a fixed list of 11 selectors that existed at audit time. `.account-delete-confirm` (introduced by 008) is not in that list, since it didn't have an entrance animation until 008 adds one. This is noted as acceptable in 008's plan — under reduced motion, the panel will still fall back to the blanket instant-snap behavior, which is correct baseline behavior, just not the gentler opacity-fade 004 introduces elsewhere. If you want full consistency, after executing both 004 and 008, manually append `.account-delete-confirm` to 004's selector list — this is optional polish, not required for either plan to be considered done.
- No other cross-plan dependencies. 002, 003, 005, and 006 are fully independent of every other plan and of each other.

## Notes

- All 8 plans were written against a codebase with no motion library (plain CSS only) and no existing modal/dropdown/toast components — the entire motion surface lives in `frontend/src/styles/global.css` plus a handful of component files.
- Every plan is self-contained per the `improve-animations` template — each can be handed to a fresh agent (via `improve-animations execute <plan>` or any other executor) with zero additional context.
